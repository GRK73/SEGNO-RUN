# Plan-B RHYTHM 구현 계획서

본 문서는 `README.md`의 기획서를 바탕으로 한 단계별 기술 구현 계획을 담고 있습니다.

## 1. 개발 환경 및 아키텍처 설정
- **기반 기술:** Vite, React, TypeScript, PixiJS, Web Audio API.
- **프로젝트 구조:**
  - `src/core`: PixiJS 메인 루프, 렌더러, 오디오 매니저.
  - `src/components`: React 기반 UI 컴포넌트 (로비, 설정, 결과창).
  - `src/game`: 게임 로직 (노트 매니저, 판정 시스템, 캐릭터 상태 관리).
  - `src/assets`: 이미지, 사운드, 채보 JSON 데이터.
- **배포:** GitHub Actions를 이용한 GitHub Pages 자동 배포 설정.

## 2. 1단계: 프로젝트 스캐폴딩 및 기초 엔진 (Week 1)
- [ ] Vite + React + TS 프로젝트 생성.
- [ ] PixiJS 애플리케이션 초기화 및 React 컴포넌트 연결 (`PixiCanvas`).
- [ ] `requestAnimationFrame` 기반의 메인 게임 루프 구축.
- [ ] 기본적인 오브젝트 풀링 시스템 (노트, 이펙트용) 설계.

## 3. 2단계: 입력 및 캐릭터 스위칭 시스템 (Week 1-2)
- [ ] **비대칭 입력 리스너:**
  - 하단 레인: 키보드 `F`, `J` 입력 처리.
  - 상단 레인: 마우스 좌/우 클릭 처리.
- [ ] **마우스 휠 스위칭:**
  - `wheel` 이벤트 캡처 및 `preventDefault()` 적용.
  - 가변 로스터 기반 캐릭터 순환 로직 (Wheel Up/Down).
  - 휠 입력 디바운싱 처리를 통한 정확한 1틱 전환 보장.

## 4. 3단계: 오디오 싱크 및 채보 시스템 (Week 2-3)
- [ ] Web Audio API를 활용한 정밀한 오디오 재생 및 현재 시간(ms) 추적.
- [ ] JSON 채보 데이터 로더 및 파서 구현.
- [ ] 시간에 따른 노트 생성 및 횡스크롤 이동 로직.
- [ ] **이중 판정 시스템:**
  - 입력 장치 일치 여부 (상단/하단).
  - 활성 캐릭터 일치 여부 (`characterId`).

## 5. 4단계: 시각적 피드백 및 UI 개발 (Week 3-4)
- [ ] **인게임 UI:**
  - 비대칭 기어 디자인 적용.
  - 휠 덱(Wheel Deck) 원형 인디케이터 (캐릭터 전환 시각화).
  - 스위칭 예고선 (Warning UI) 구현.
- [ ] **아웃게임 UI:**
  - 곡 선택 리스트, 로딩 화면, 결과 정산 화면.
  - 키 바인딩 및 상/하단 반전 설정 메뉴.

## 6. 5단계: 폴리싱 및 최적화 (Week 4+)
- [ ] 델타 타임(Delta Time) 기반 이동 보정으로 프레임 드랍 대응.
- [ ] 에셋 프리로딩 시스템 구축.
- [ ] GitHub Pages 배포 및 모바일 브라우저 차단(또는 경고) 처리.

## 7. 주요 기술적 도전 과제
- **휠 입력 정밀도:** 브라우저/OS마다 다른 휠 델타 값을 정규화하여 일정한 조작감 제공.
- **오디오 레이턴시:** `Web Audio API`를 사용하여 시각적 요소와 오디오의 싱크를 16ms 이내로 유지.
- **가변 로스터 확장성:** 1인~4인 등 곡마다 다른 캐릭터 구성을 유연하게 처리하는 상태 머신 설계.
